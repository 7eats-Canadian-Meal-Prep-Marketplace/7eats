# Waitlist Backend — Design Spec

**Date:** 2026-05-14
**Project:** 7eats — Marketplace Cooks
**Scope:** API + database only. No frontend, no email sending.

---

## Context

7eats is a two-sided food marketplace connecting home cooks with customers seeking affordable, culturally diverse meal prep — launching first in Montreal and Toronto/Hamilton. This spec covers the waitlist signup backend: a single API endpoint that captures email addresses from interested parties before launch.

## Database Schema

Two tables, versioned via Drizzle migrations.

### `waitlist`

| Column | Type | Constraints |
|---|---|---|
| `id` | UUID | Primary key, default `gen_random_uuid()` |
| `email` | text | Unique, not null |
| `ip_hash` | text | Not null |
| `created_at` | timestamptz | Default `now()` |

Stores one row per unique email. Duplicate submissions are silently ignored via upsert conflict handling — the table stays clean.

### `rate_limit_log`

| Column | Type | Constraints |
|---|---|---|
| `id` | UUID | Primary key, default `gen_random_uuid()` |
| `ip_hash` | text | Not null, indexed |
| `attempted_at` | timestamptz | Default `now()` |

A row is inserted here on **every inbound request** before any other logic executes. This ensures all attempts — new signups, duplicate retries, and bot probes — count toward the rate limit. Index on `(ip_hash, attempted_at)` makes the count query fast and supports future pruning.

## API

### `POST /api/waitlist`

#### Guards (checked before any DB interaction)

1. `Content-Type: application/json` must be present
2. Body must be ≤ 1 KB
3. `User-Agent` header must be present and must not match known bot patterns (e.g. `curl`, `python-requests`, `scrapy`, empty string)

Any guard failure returns `400`.

#### Request Body

```json
{ "email": "user@example.com" }
```

Validated with Zod. `email` is required and must be a valid email address. No other fields are accepted.

#### Request Flow

1. Run guards (Content-Type, body size, User-Agent)
2. Hash the request IP with SHA-256 → `ip_hash`
3. Insert a row into `rate_limit_log`
4. Count rows in `rate_limit_log` for this `ip_hash` within the rolling window
5. If count exceeds `RATE_LIMIT_MAX_ATTEMPTS`, return 429
6. Parse and validate body with Zod; return 400 on failure
7. Upsert email into `waitlist` (ignore unique conflict)
8. Return 200

#### Response Envelope

All responses follow `{ success: boolean, message: string }`.

| Scenario | Status | `success` | `message` |
|---|---|---|---|
| New signup | 200 | `true` | `"You're on the list!"` |
| Duplicate email | 200 | `true` | `"You're on the list!"` |
| Rate limited | 429 | `false` | `"Too many attempts. Try again later."` |
| Invalid body | 400 | `false` | `"Invalid request."` |
| Bot / missing UA | 400 | `false` | `"Invalid request."` |
| Server error | 500 | `false` | `"Something went wrong."` |

Duplicate email and new signup return identically — no information leakage about whether an email exists.

Internal errors are logged server-side only. Stack traces and DB details never appear in responses.

## Security

- Raw IPs are never stored. All IPs are hashed with SHA-256 before any persistence.
- Rate limiting is enforced on all attempts, not just successful ones.
- Idempotency: submitting the same email twice is safe and returns the same 200 response.
- Bot filtering via User-Agent heuristics (basic layer; not a substitute for CAPTCHA if needed later).

## Environment Variables

| Variable | Required | Default | Purpose |
|---|---|---|---|
| `DATABASE_URL` | Yes | — | Neon Postgres connection string |
| `RATE_LIMIT_WINDOW_MINUTES` | No | `60` | Rolling window length in minutes |
| `RATE_LIMIT_MAX_ATTEMPTS` | No | `3` | Max attempts per IP per window |

## File Structure

All files inside `my-app/`:

```
my-app/
├── app/
│   └── api/
│       └── waitlist/
│           └── route.ts          # Thin orchestrator
├── db/
│   ├── index.ts                  # Neon + Drizzle client singleton
│   ├── schema.ts                 # Drizzle table definitions
│   └── migrations/               # Generated by drizzle-kit
├── lib/
│   ├── hash.ts                   # SHA-256 IP hashing
│   ├── rate-limit.ts             # Log attempt + count check
│   ├── validation.ts             # Zod schema + request guards
│   └── waitlist.ts               # Upsert email into waitlist
├── .env.example                  # Documented variable template
└── drizzle.config.ts             # drizzle-kit config
```

### Module Responsibilities

| Module | Does exactly one thing |
|---|---|
| `lib/hash.ts` | Accepts a raw IP string, returns a SHA-256 hex digest |
| `lib/rate-limit.ts` | Inserts a log row; queries count vs. window; returns allow/deny |
| `lib/validation.ts` | Exports Zod schema + request guard function |
| `lib/waitlist.ts` | Upserts an email into the waitlist table |
| `db/schema.ts` | Drizzle table definitions only |
| `db/index.ts` | Creates and exports the Drizzle client (singleton pattern) |
| `app/api/waitlist/route.ts` | Orchestrates the above; owns request/response shape |
